# Task 27 — 최초 Naver API Token 발급 테스트 실행 계획 문서

이 문서는 실제 Naver API 토큰 발급 코드를 작성하기 전에, 공식 문서를 검토하고 안전한 테스트 실행 계획을 수립하기 위해 작성되었습니다. 이 문서 작성 시점까지 어떠한 실제 토큰 발급이나 상품 API 호출도 실행되지 않았습니다.

---

## 1. 현재 안전 구조 요약

지금까지 아래의 안전 장치들이 모두 구현되고 검증되었습니다.
* **Auth Config Safe Reader skeleton 완료**: 설정된 인증 정보의 안전한 읽기 전용 검증
* **Token Provider disabled skeleton 완료**: 토큰 제공자의 기본 상태를 비활성화로 고정
* **Token Dry Permission Gate 완료**: 실제 실행 전 권한 검증 및 차단
* **Token Provider test-only skeleton 완료**: 실행 환경을 분리하여 안전한 테스트 환경 구성
* **Token Test Approval Audit 완료**: 테스트 전 필수 확인 사항들에 대한 사용자 동의 기록 저장
* **Token First Test Safety Boundary 완료**: 15개 섹션의 체크리스트를 통과해야만 테스트 준비 완료로 판정
* **실제 token 발급**: 0회
* **실제 Naver API 호출**: 0회
* **민감 정보 상태**: endpoint URL, HTTP client, authorization header 없음

---

## 2. 공식 문서 확인 결과

네이버 커머스 API 공식 문서를 통해 확인한 인증 토큰 발급의 주요 원칙은 다음과 같습니다.

*   **토큰 발급 방식**: OAuth 2.0의 `client_credentials` 방식을 사용합니다.
*   **필요한 Credential**: 
    *   `client_id` (애플리케이션 ID)
    *   `timestamp` (밀리초 단위 Unix 시간)
    *   `client_secret_sign` (클라이언트 시크릿을 기반으로 한 전자서명: `client_id` + `timestamp`를 `client_secret`으로 bcrypt 해싱 후 Base64 인코딩)
    *   보안을 위해 `client_secret` 원문은 전송하지 않고 서명 생성에만 사용됩니다.
*   **API 분리 원칙**: 토큰 발급 요청(공식 인증 endpoint)과 실제 리소스(상품 수정 등) 호출 endpoint는 완전히 분리되어 독립적으로 호출할 수 있습니다.
*   **토큰 유효기간**: 발급된 토큰의 유효 시간은 3시간(10,800초)입니다.
*   **실패 처리 원칙**: API 호출 시 인증 에러(`401 Unauthorized`)가 발생하면 토큰이 만료되거나 유효하지 않은 것이며, 즉시 재발급 절차를 거쳐야 합니다.
*   **토큰 보안**: 발급받은 토큰은 `Authorization: Bearer <인증토큰>` 형태로 사용되며, 외부에 노출되거나 영구 저장소에 보관해서는 안 됩니다.

---

## 3. 최초 Token 발급 테스트 범위

이번 테스트는 인증 메커니즘의 정상 동작 여부만 검증하는 가장 좁은 범위의 단발성 테스트입니다.

*   **요청 횟수**: 단 1회의 토큰 발급 요청만 실행합니다.
*   **상품 API 호출 차단**: 상품 조회, 등록, 수정 등 어떠한 상품 관련 API도 호출하지 않습니다.
*   **시스템 분리**: 비동기 Queue나 Worker로 작업을 넘기지 않고, 즉각적인 응답 상태만 확인합니다.
*   **저장 금지**: 응답받은 토큰은 메모리에 임시 보관되며 파일, DB 등에 저장하지 않습니다.
*   **노출 금지**: 토큰 원문이나 헤더는 로그 및 UI에 출력하지 않습니다.
*   **결과 기록**: 오직 `safe status`(성공/실패 등 추상화된 상태 코드)만 기록합니다.
*   **단절된 흐름**: 토큰 발급에 성공하더라도 자동으로 상품 수정이나 다음 단계로 진행되지 않습니다.

---

## 4. Token 처리 원칙

보안 사고를 방지하기 위해 다음 원칙을 어떠한 예외도 없이 강제합니다.

*   **출력 금지**: `access_token`, `refresh_token`, `authorization` 헤더 원문 로그/UI 출력 절대 금지
*   **저장 금지**: 토큰 자체 및 관련된 메타데이터의 DB/파일 저장 절대 금지
*   **마스킹 금지**: UI 상에 토큰의 마스킹된 형태(예: `ab***xyz`)조차 표시하지 않음
*   **단기 생명주기**: 토큰은 변수에 할당되어 검증 로직 내에서만 잠깐 존재하고 즉시 가비지 컬렉션 되도록 설계
*   **간접 표시**: 테스트 성공 여부는 `NAVER_AUTH_TOKEN_FIRST_TEST_SUCCESS` 와 같은 안전한 상태 코드로만 표시
*   **응답 본문 통제**: 토큰이나 인증 관련 민감한 정보가 포함될 수 있는 raw request body 및 raw response body의 저장을 전면 차단

---

## 5. 실제 Token 발급 테스트 전 필수 조건

테스트 코드가 실행되기 위해서는 다음 조건들이 모두 만족되어야 합니다.

*   **환경 상태**: `origin/main` 최신 상태 동기화 및 작업트리 `clean` 상태
*   **Safety Boundary**: Task 26에서 구현한 Safety Boundary가 정상적으로 존재하고, 조건 검사를 통과할 것
*   **플래그 상태**:
    *   `readyForExplicitTokenTestApproval=true` 여부 확인
    *   `allowed=false`, `tokenRequestAllowed=false` 상태가 안전하게 유지되고 있는지 확인
*   **Audit 상태**: 모든 필수 항목에 대해 동의한 Token Test Approval Audit 기록이 존재할 것
*   **최종 승인**: 시스템의 자동 실행이 아닌 **사용자의 별도 명시 승인**이 이루어졌을 것
*   **논리적 격리**: 토큰 발급 Task가 상품 수정 endpoint 호출 가능성과 완벽히 분리될 것
*   **기타 차단**: Live 실행 비활성화, Queue/Worker 연결 차단이 유지될 것

---

## 6. 사용자 최종 승인 문구

테스트 실행 직전 사용자에게 제시하고 동의를 받아야 할 명시적인 승인 문구입니다.

> **“최초 Naver API token 발급 테스트 1회를 승인합니다. 이 테스트는 상품 수정 API 호출, 상품 조회 API 호출, Queue/Worker 실행, Live 실행과 연결하지 않으며, token 원문을 저장·출력·로그·UI 표시하지 않습니다.”**

---

## 7. 실패 처리 계획

요청이 실패할 경우, 시스템의 안전을 최우선으로 보호합니다.

*   **자동 재시도 금지**: 네트워크 오류, 401 권한 오류 등 어떠한 오류에도 시스템이 자동으로 재시도하지 않습니다.
*   **안전한 에러 로깅**: 실제 API에서 반환된 raw error response를 저장하지 않으며, `NAVER_API_TOKEN_REQUEST_FAILED`와 같은 사전 정의된 safe error code만 기록합니다.
*   **본문 차단**: 에러 응답 본문에 서명이나 시크릿 관련 정보가 포함될 수 있으므로 저장을 차단합니다.
*   **흐름 중단**: 실패 시 어떠한 경우라도 상품 수정 흐름 등 다음 단계로 이행되지 않습니다.

---

## 8. 성공 처리 계획

요청이 성공할 경우, 토큰이 노출되지 않도록 엄격히 통제합니다.

*   **노출 원천 차단**: 응답에서 토큰을 추출하더라도 이를 로그, 터미널, 파일, DB 어디에도 원문으로 저장/출력하지 않습니다.
*   **상태 기록**: 토큰의 존재 여부와 파싱 가능 여부만 판단한 뒤, 성공했다는 safe status만 기록하고 토큰 데이터는 즉시 폐기합니다.
*   **실행 단절**: 토큰 발급 성공이 상품 API 호출, Live 실행, Queue/Worker 스케줄링으로 이어지지 않도록 합니다.
*   **수동 진행**: 성공 이후의 단계는 전적으로 별도의 Task 기획과 추가적인 사용자 승인을 통해서만 진행 가능합니다.
